# AI 规则文件

## 版本管理规则

### 1. 版本号格式
- 使用语义化版本号格式：`vX.Y.Z`
- X：主版本号，当进行不兼容的API更改时递增
- Y：次版本号，当添加功能但保持向后兼容时递增
- Z：补丁版本号，当进行向后兼容的错误修复时递增

### 2. 版本更新流程
1. **代码更新**：完成代码修改后，更新版本号
2. **版本文件**：在适当的位置（如 `build.gradle` 或版本配置文件）更新版本号
3. **Git 提交**：提交代码更改
4. **Git Tag**：创建与版本号对应的标签，格式为 `vX.Y.Z`

### 3. Git 操作流程
1. 确保所有更改已添加到暂存区：`git add .`
2. 提交更改，包含版本更新信息：`git commit -m "Update version to vX.Y.Z"`
3. 创建版本标签：`git tag -a vX.Y.Z -m "Version vX.Y.Z"`
4. 推送提交和标签：`git push && git push --tags`

### 4. 版本更新触发条件
- **功能添加**：添加新功能时，递增次版本号（Y），重置补丁版本号（Z）为0
- **Bug 修复**：修复错误时，递增补丁版本号（Z）
- **重大更改**：进行不兼容的API更改时，递增主版本号（X），重置次版本号（Y）和补丁版本号（Z）为0

### 5. 示例
- 从 v1.3.2 修复bug → v1.3.3
- 从 v1.3.3 添加新功能 → v1.4.0
- 从 v1.4.0 进行重大更改 → v2.0.0

## 代码规范

### 1. 编码风格
- 遵循 Kotlin 官方编码规范
- 使用 4 个空格进行缩进
- 类名使用 PascalCase
- 方法和变量名使用 camelCase
- 常量使用 UPPER_SNAKE_CASE

### 2. 提交信息规范
- 提交信息应清晰、简洁，描述本次更改的内容
- 使用英文撰写提交信息
- 提交信息格式：`[类型] 简短描述`
- 类型包括：feat（功能）、fix（修复）、docs（文档）、style（样式）、refactor（重构）、test（测试）、chore（其他）

### 3. 分支管理
- `main`：主分支，用于发布稳定版本
- `develop`：开发分支，用于集成新功能
- `feature/*`：功能分支，用于开发特定功能
- `bugfix/*`：修复分支，用于修复特定bug

## 自动化建议

### 1. 版本更新自动化
- 考虑使用 Gradle 插件或脚本自动管理版本号
- 在构建过程中自动生成版本信息

### 2. Git 操作自动化
- 配置 Git hook 以在提交时自动检查版本更新
- 使用脚本简化版本更新和标签创建流程

### 3. CI/CD 集成
- 在 CI/CD 流程中集成版本管理
- 自动构建和发布对应版本的应用
